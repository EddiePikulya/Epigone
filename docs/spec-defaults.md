# Spec defaults (carried into /to-spec)

Working defaults agreed during the 2026-07-09 grilling session. Not architecture (see `docs/adr/`), not glossary (see `CONTEXT.md`) ‚Äî tunable starting values for the spec.

- **Stream poll interval: 30 seconds** (user decision 2026-07-09). Capacity at 30s ‚âà 110 distinct tracked wallets (weight 4 per wallet per poll ‚Äî core + xyz venues ‚Äî so 8 weight/min/wallet against the shared budget). Global trades-feed firehose removes the ceiling when needed.
- **Rate budget: one shared 900 weight/min bucket** (issue #28, 2026-07-10). Ingest + stream draw from a single Postgres-backed token bucket ‚Äî 75% of Hyperliquid's 1200/min per-IP cap, i.e. 25% steady-state headroom; burst capped at 240 so even the worst rolling minute (full burst + a minute of refill = 1140) stays under the cap. The stream has priority: ingest never draws the bucket below a 120-weight reserve floor. The gateway backs off and retries on 429 (`Retry-After`, else exponential + jitter); a persistent streak surfaces as `RateLimitedError`, which passes treat as pacing ‚Äî retry later, never abort.
- **Rate calibration** (issue #41, 2026-07-11). Two additions to the shared bucket eliminated the residual 429 trickle. (1) **Real fills weight**: per HL's rate-limit docs, `userFills` costs 20 *plus* weight per 20 fills returned (up to ~+100 on a full ~2000-fill response) ‚Äî the root cause of 429s at a nominally under-budget ~600/min. The fine pass settles the surcharge (`ceil(fills/20)`, `FILLS_PER_SURCHARGE_WEIGHT`) against the budget post-response via `Budget.settle`, which may drive the bucket into debt that later spends pace off. (2) **Send gate** (`SMOOTHING_WEIGHT_PER_SECOND = 20`, i.e. the cap spread uniformly per second): every grant reserves a window proportional to its weight, so the bucket can't be emptied in a sub-second spike; the gate bounds a stream poll's worst wait to one ingest call's window plus its settled surcharge (‚âà6s for a full ~2000-fill response ‚Äî well inside the 30s poll interval), keeping the reserve-floor priority intact. Observability: the budget row meters real consumed weight and logs actual-vs-limit each minute (`epigone.budget`). Note: HL's `userRateLimit` info endpoint was probed (2026-07-11) and reports *address-based action* limits (`nRequestsUsed`/`nRequestsCap`), not IP info-request weight ‚Äî it always reads 0 for us, hence the budget-side metering instead. Constants are starting values; final tuning is empirical against the production server's IP after merge.
- **Metrics refresh tiers:** full-universe scan is an ~11+ hour rate-limited batch (run rarely); active/tracked/screener-visible traders refresh incrementally (hourly‚Äìdaily); dormant accounts swept weekly.
- **Two-stage scan (discovered via Bullpen R&D 2026-07-09):** HL's info API `portfolio` request returns precomputed windowed PnL / account value / volume (day, week, month, allTime, perp variants) for any address in a single weight-20 call ‚Äî verified live via `bullpen hyperliquid stats --address`. Stage 1 (coarse): portfolio stats across the Universe ‚Üí cheap windowed PnL/ROI/volume metrics, one call per account, no fill paging. Stage 2 (fine): fills-based metrics (win rate, avg win/loss, Sharpe, drawdown) only for accounts passing coarse filters (likely a few thousand). Screener filters should distinguish coarse metrics (available Universe-wide) from fine metrics (available for the vetted subset).
- **Fine-metrics backfill via S3 (ecosystem survey 2026-07-09):** Hyperliquid publishes raw fills to official S3 buckets (`hl-mainnet-node-data/node_fills_by_block`, `hyperliquid-archive`; free third-party mirror: Reservoir). Prefer bulk-downloading fills and computing fine metrics offline for the whole Universe over rate-limited API paging; the API budget then serves only incremental freshness + the stream. Verify bucket access/cost/lag at build time.
- **Alert noise controls** (issue #10): a **minimum position size** (`users.min_size_usd` global, `tracks.min_size_usd` per-Track override) suppresses alerts for positions notionally smaller than the floor, across every alert kind; **mute** (`tracks.muted`) silences a Track without unfollowing. Both are edited from `/tracked`. Suppression happens at queue time in the stream poller, never at delivery ‚Äî so unmuting or raising a floor never dumps a backlog of the events missed while suppressed.
- **Alert events:** position open / close / flip; **scale-in / scale-out** on a same-coin/same-side size change (issue #10); on-demand current-positions view.
- **Scale-in/out significance threshold: 25%** (`SCALE_SIGNIFICANCE_THRESHOLD`, `epigone.stream.poller`). A same-side notional size change of ‚â•25% of the last snapshot fires SCALE-IN (bigger) or SCALE-OUT (a partial decrease that isn't a full close); smaller drift stays a silent snapshot update, preserving the pre-#10 partial-close/add behaviour. Conservative by design ‚Äî over a 30s poll a real coin never moves 25% on price alone, so the signal reflects a deliberate add/trim, not mark-price noise. Change is measured against the last snapshot each poll, so gradual accumulation that never clears 25% in one step stays quiet.
